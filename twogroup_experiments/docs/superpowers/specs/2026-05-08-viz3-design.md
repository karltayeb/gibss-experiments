# 2026-05-08 Viz3 Design

**Goal:** Build a new visualization notebook, `notebooks/viz3.py`, with a clean companion module, `viz3_utils.py`, that does not depend on `viz2` and can progressively restore all non-placeholder plot families from the current notebook.

## Scope

`viz3` replaces the current `viz2` workflow for interactive plotting of prepared experiment outputs. It must start as a minimal shell, then grow plot-by-plot until it supports all currently real plots:

1. PIP calibration
2. Power vs FDP
3. Causal PIP vs threshold
4. Credible set power/size/coverage summary
5. Credible set scenario points
6. Credible set histograms

The unconditional credible-set coverage section in `viz2` is explicitly out of scope because it is placeholder-only today.

`viz3` must not import from `viz2.py`, `viz2_logic.py`, or `viz2_metadata.py`. If logic is worth keeping, it should be reimplemented cleanly in `viz3_utils.py`.

## Files And Responsibilities

- `notebooks/viz3.py`
  Thin Marimo notebook shell. Owns page structure, section headings, tabs, and UI controls. Calls `viz3_utils` for loading, filtering, summarization, and chart rendering.

- `viz3_utils.py`
  Pure Python helpers for:
  - collection discovery and manifest loading
  - parquet loading and empty fallback tables
  - method metadata parsing and method selection helpers
  - plot-specific data preparation
  - statistical summaries
  - matplotlib chart rendering

- `tests/test_viz3_utils.py`
  Unit coverage for clean-room logic in `viz3_utils.py`, starting with shared loading/filtering and first end-to-end plot path.

## Product Shape

The notebook should feel like one coherent tool instead of many copied sections. Structure:

1. Collection selection
2. Shared method selection controls
3. Plot family sections, each in its own panel/tab group

Initial shell should show all real plot families so user can see target structure immediately, but only completed plot families should render real charts. Incomplete plot families may render an explicit “Not implemented in viz3 yet” placeholder during staged delivery.

## Shared Data Model

`viz3_utils.py` should expose a single loader entry point that returns one structured bundle for a collection root. Exact representation can be a dataclass, typed dict, or simple namespace, but it must contain:

- collection manifest metadata
- `pip_threshold_plot_data`
- `causal_pip_plot_data`
- `cs_component_plot_data`
- `cs_truth_plot_data`

Each table should be loaded independently from the collection root using the existing file layout:

- `batches/*/fits/*/pip_threshold_plot_data.parquet`
- `batches/*/fits/*/causal_pip_plot_data.parquet`
- `batches/*/fits/*/cs_component_plot_data.parquet`
- `batches/*/fits/*/cs_truth_plot_data.parquet`

If a dataset is missing, loader returns an empty dataframe with stable schema instead of failing.

All loaded tables should be normalized once up front by adding method metadata columns:

- `method_family`
- `L`
- `is_thresholded`
- `is_oracle`
- `method_label_base`
- `method_display`

## Shared Control Model

Shared controls should be defined once and reused across sections:

- collection selector
- method family selector
- `L` selector
- derived selected methods set

Plot-local controls remain local:

- `threshold`
- `max_fdp`
- `fixed_y_scale`
- `show_background_threshold_traces`
- `nominal_coverage`
- `max_cs_size`
- `min_ser_log_bf`

Important simplification: method selection should have one source of truth. The notebook should derive selected methods from shared selectors via utility helpers rather than duplicating filtering code in each section.

## Stage Delivery

### Stage 1: Minimal Shell

Deliver a new `viz3.py` notebook that:

- loads collection options
- imports `viz3_utils`
- instantiates shared controls
- displays the six real plot families in final notebook order
- uses temporary explicit placeholders for incomplete plot families

This stage validates notebook structure and module boundaries without reintroducing `viz2` sprawl.

### Stage 2: PIP Calibration End-To-End

Implement the first fully working plot family:

- data loading from collection
- threshold filtering
- method filtering
- calibration binning
- aggregate chart
- by-simulation chart

This is the first full vertical slice because it uses one dataset and simpler transformations than the credible-set plots.

### Stage 3: Shared Loader Expansion

Once calibration works, loader and metadata helpers become the canonical foundation for all remaining plots. Add or refine only the abstractions needed by the next plot family; avoid speculative framework code.

### Stage 4+: Remaining Plot Families

Add plot families incrementally in this order:

1. Power vs FDP
2. Causal PIP vs threshold
3. Credible set power/size/coverage summary
4. Credible set scenario points
5. Credible set histograms

This order moves from simple single-table summaries to more complex credible-set joins and derived summaries.

## Architecture Rules

1. `viz3.py` must stay thin.
   No heavy dataframe logic, metadata parsing, bootstrap code, or matplotlib series construction should live in notebook cells unless it is purely presentational.

2. `viz3_utils.py` should expose plot-family functions with consistent shape.
   Preferred pattern:
   - prepare data
   - summarize data
   - render chart

3. Shared helpers must be generic only where already justified by two or more plot families.
   Avoid abstracting ahead of need.

4. Placeholders in `viz3` are allowed only for staged, not-yet-implemented plot families.
   Final target is zero placeholder-only sections among in-scope plots.

## Error Handling

- Missing collection or missing parquet files should render empty-state charts instead of tracebacks.
- Unknown method metadata should degrade to raw method names rather than failing.
- Empty filtered datasets should preserve schema and remain renderable.

## Testing Strategy

`viz3_utils.py` should be unit-tested outside Marimo. Tests should focus on:

- collection loading with missing-data fallback
- metadata extraction from `method_spec`
- method-family and `L` selection
- PIP calibration preparation and summary
- later, each added plot family’s core summary logic

Notebook smoke testing should confirm `uv run python notebooks/viz3.py` imports cleanly and Marimo can parse notebook structure without requiring live user interaction.

## Non-Goals

- Perfect UI polish in first stage
- Backward compatibility with `viz2` internals
- Refactoring `viz2` itself
- Implementing unconditional CS coverage before real use case exists

## Recommendation

Build `viz3` as a clean-room replacement, not a migration layer. The right simplification is stronger boundaries:

- notebook = controls and layout
- utils = data + plotting logic

That gives a stable foundation for staged delivery without carrying forward `viz2`’s mixed responsibilities.
